présentation · 2
présentation

un dépôt d'orchestration, pas de code applicatif

le meta-repo pilote le recode multi-techno sur les dépôts d'un client. il porte la configuration et les contrats, génère les plans et collecte les sorties. les skills, eux, sont installés depuis un dépôt amont et personnalisés par des overrides versionnés.

configuration
racine (workspace.yaml, context.yaml, CLAUDE.md) et contrats dans specs/.
plans générés
le DAG d'exécution produit par le planner : un rendu Markdown daté et plan-last.yaml.
sorties
doc de synthèse, stack-snapshot.yaml, wiki LLM et journaux d'action par repo.
# structure du dépôt meta-repo/ ├── workspace.yaml # repos + chemins ${REPOS_ROOT} ├── context.yaml # politiques + DoD mesurable ├── CLAUDE.md # contexte + garde-fous ├── scripts/ # init · résolution · scan · run · exec ├── specs/ # dod.md + guidelines/ ├── skills-overrides/ # personnalisations client ├── plans/ · docs/ · logs/ # sorties └── .claude/skills/ # skills matérialisés
i
frontière meta / sous-repo. on ne commit jamais de code applicatif depuis la racine. les changements applicatifs vivent dans les sous-dépôts ; les commits du meta-repo concernent uniquement le scaffolding (config, scripts, docs, .claude/).

concepts clés · 3
concepts clés

six idées structurantes

à comprendre avant tout usage du meta-repo.

configuration portable
les chemins utilisent ${REPOS_ROOT}. aucune donnée machine dans le versionné : chacun fournit son workspace.local.yaml.
init idempotente
en deux phases : placeholders et config, puis clone des repos et détection des stacks. ré-exécutable sans danger.
skills amont + overrides
matérialisés par sync_skills.py ; les overrides client versionnés survivent aux mises à jour amont.
validation tolérante
un repo non cloné est ignoré par défaut. --strict exige tous les repos présents pour la CI.
wiki LLM automatique
les éditions de docs de synthèse déclenchent l'ingestion via des hooks. aucune invocation manuelle requise.
journaux par repo
chaque action sur un sous-repo est consignée dans logs/<repo>.md en append-only.

le pipeline en 6 couches

de l'intention brute à la pull request, en passant par deux gates bloquants.

1
analysegate spec.validatedgate toolchain_verified
ticket-fetcher · clarification-loop · intent-parser · repo-resolver · tech-detector · toolchain-preflight
2
orchestration
planner construit le DAG de tâches et sélectionne les sous-skills techno
3
exécution
code-reader · diff-writer (code métier) · test-generator (tests)
4
review
code-reviewer produit un rapport structuré et route le retry
5
validation
validator enchaîne lint, build, tests et retry dans le conteneur déclaré
6
livraison
pr-builder crée des branches et des PRs neutres, cross-référencées
!
deux gates bloquants. clarification-loop ne laisse pas démarrer le pipeline tant que spec.validated est faux ; toolchain-preflight suspend avant le planner si le runtime ou le gestionnaire de paquet ne sont pas à la version épinglée.

exécution · 7
exécution conteneurisée

chaque commande dans le bon conteneur

version, build, lint, test, coverage : un point d'exécution central enveloppe chaque commande d'un repo selon son bloc exec, vers un service docker-compose ou l'hôte. plus aucune dépendance aux toolchains installées localement.

bloc exec par repo
déclaré dans workspace.yaml, surchargeable dans workspace.local.yaml : mode, service, compose_file, workdir.
point d'exécution central
repo_exec.py (et repo-exec.sh) route toute commande vers docker compose exec ou l'hôte. jamais d'appel direct à mvn / npm.
deux modes
compose exécute dans un service déjà lancé ; native (défaut) sur l'hôte. le gate --check vérifie docker et le service avant de lancer.
# workspace.yaml — bloc exec repos: src-api: exec: mode: compose compose_file: ~/infra/dc.yml # hors du repo service: api workdir: /workspace/src-api # projet dans le conteneur # lancer une commande dans le conteneur repo-exec.sh src-api -- mvn test
i
routage automatique. toolchain-preflight et validator passent par ce point : checks de version et build/test s'exécutent dans le conteneur déclaré. compose_file peut vivre hors du repo (~, $VAR) et workdir cible le projet quand un même conteneur en héberge plusieurs.

cartographie · 8
cartographie cross-repo

graphify — un graphe de tout le code

chaque repo devient un graphe de connaissances (god nodes, communautés, relations inter-fichiers). extraction ast-only : déterministe, sans llm ni clé api, reproductible et hors-ligne.

initialisation
le skill /graphify est livré par meta-repo-resources et matérialisé par sync_skills.py (étape 2). le lanceur scripts/graphify-map.sh est versionné dans ce meta-repo. le cli graphify reste un prérequis sur le PATH.
lancement
graphify-map.sh résout workspace.yaml et génère le graphe de chaque repo cloné. le skill /graphify interroge le graphe : query, explain, path.
sorties
graphify-out/ dans chaque repo cloné (graph.json, graph.html, GRAPH_REPORT.md). cache/ et .graphify_root gitignorés.
# matérialiser le skill puis cartographier python3 scripts/sync_skills.py scripts/graphify-map.sh # tous les repos clonés scripts/graphify-map.sh --repo src-esp # interroger la cartographie (/graphify) graphify query "comment les stats sont calculées ?" graphify explain "AccountStats"
i
extraction ast-only, hors-ligne. aucun appel llm ni clé api : le graphe est déterministe et reproductible. graphify-out/ est régénérable à tout moment via scripts/graphify-map.sh.
